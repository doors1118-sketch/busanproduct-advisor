# 공공계약 법령 해석 챗봇 서비스 개발 계획

## 1. 개요

부산광역시 공공조달 모니터링 시스템의 부가 서비스로, 계약 담당자가 법령·예규·내규에 대해 질문하면 **근거 조항과 함께 해석을 제공**하는 AI 챗봇.

```
사용자 질문 → 관련 법령 검색(RAG) → Gemini가 교차 해석 → 답변 + 근거 조항
```

---

## 2. 데이터 계층 구조

| 계층 | 문서 유형 | 수집 방법 | 업데이트 |
|:---:|------|------|------|
| L1 | 지방계약법·시행령·시행규칙 | **법령 API 자동** | 주 1회 |
| L2 | 행안부 계약예규 (입찰·계약 집행기준) | **법령 API 자동** | 주 1회 |
| L3 | 조달청 입찰 조건·고시·기준 | API 또는 반자동 | 월 1회 |
| L4 | 부산시 자체 계약 지침 | 관리자 업로드 | 수동 |
| L5 | 자치구군·출자출연기관 내규 | 관리자 업로드 | 수동 |
| L6 | 행안부 질의회신·감사원 결정례 | 수동 수집 | 분기 |

> [!IMPORTANT]
> 답변 시 **L1(상위법) → L2~3(예규) → L4~5(내규)** 순서로 근거를 표시하여 법적 우선순위가 명확하도록 설계

---

## 3. 기술 스택

| 구분 | 선택 | 사유 |
|------|------|------|
| **LLM** | Gemini (최신 모델) | 기존 구독 활용, 롱컨텍스트 |
| **Embedding** | Gemini text-embedding | 동일 생태계, 한국어 지원 |
| **벡터 DB** | ChromaDB | 경량, Python 네이티브, 로컬 실행 |
| **프레임워크** | LangChain | RAG 파이프라인 표준, Gemini 연동 내장 |
| **프론트엔드** | Streamlit | 기존 대시보드 기술, 챗 UI 내장 |
| **법령 수집** | 국가법령정보센터 OpenAPI | 법령·행정규칙 전문 제공 |
| **배포** | 기존 NCP 서버 | 추가 인프라 불필요 |

## 4. 시스템 아키텍처

```
┌─────────────────────────────────────────────────────┐
│                    데이터 수집 레이어                  │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────┐ │
│  │ 법령 API     │  │ 예규·고시    │  │ 내규 업로드 │ │
│  │ (L1~L2 자동) │  │ (L3 반자동)  │  │ (L4~L5)    │ │
│  └──────┬───────┘  └──────┬───────┘  └─────┬──────┘ │
└─────────┼─────────────────┼────────────────┼────────┘
          ▼                 ▼                ▼
┌─────────────────────────────────────────────────────┐
│                   데이터 가공 레이어                   │
│  텍스트 추출 → 조·항·호 단위 청크 분할 → 메타데이터 태깅 │
│  (법령명, 조항번호, 계층, 시행일, 소속기관)             │
└─────────────────────┬───────────────────────────────┘
                      ▼
┌─────────────────────────────────────────────────────┐
│                   벡터 DB (ChromaDB)                  │
│  Collection: laws / regulations / guidelines / qalog │
│  메타데이터 필터: 계층(L1~L6), 기관명, 분야           │
└─────────────────────┬───────────────────────────────┘
                      ▼
┌─────────────────────────────────────────────────────┐
│                   RAG 엔진 (LangChain)               │
│  ① 질문 임베딩 → ② 벡터 유사도 검색 (Top-K)          │
│  → ③ 메타데이터 필터 (기관·분야)                      │
│  → ④ 관련 조항 10~20개 추출                          │
│  → ⑤ Gemini에 질문+조항 전달 → 해석 답변 생성         │
└─────────────────────┬───────────────────────────────┘
                      ▼
┌─────────────────────────────────────────────────────┐
│              프론트엔드 (Streamlit 챗봇)               │
│  기관 선택 → 질문 입력 → 답변(해석+근거조항+원문링크)   │
│  + 대화 이력 저장 → 자주 묻는 질문 자동 생성            │
└─────────────────────────────────────────────────────┘
```

## 6. 청크 분할 전략 (핵심)

법령은 일반 문서와 다르게 **조·항·호** 구조가 명확하므로:

```
✗ 일반 청크: 500자씩 무조건 분할 → 조항이 잘릴 수 있음
✓ 법령 청크: 조 단위 분할 + 관련 항·호 포함 → 의미 단위 보존
```

| 청크 전략 | 단위 | 메타데이터 |
|----------|------|-----------|
| 법률·시행령 | 조(Article) 단위 | 법령명, 조번호, 시행일 |
| 예규·지침 | 조 또는 절(Section) 단위 | 예규명, 조번호, 시행일 |
| 질의회신 | 건(Case) 단위 | 회신일, 질의 요지, 결론 |
| 기관 내규 | 조 또는 페이지 단위 | 기관명, 문서명, 시행일 |

## 7. 보안 및 운영 고려사항

| 항목 | 방안 |
|------|------|
| 데이터 보안 | 기관 내규는 기밀 가능 → NCP 서버 내 로컬 처리, 외부 전송 최소화 |
| LLM 입력 | 내규 원문은 Gemini에 전달됨 → 민감 정보 마스킹 검토 |
| 답변 면책 | "본 답변은 참고용이며 법적 효력이 없습니다" 고지 필수 |
| 접근 제어 | 기관별 내규는 해당 기관 사용자만 조회 가능 (인증) |
| 로그 보존 | 질문·답변 이력 6개월 보관 → 품질 분석용 |

## 9. 기대 효과

- **계약 담당자 업무 효율**: 법령 검색·해석 시간 대폭 단축 (건당 30분→2분)
- **정확도 향상**: 관련 조항 교차 검토로 누락·오류 방지
- **지식 표준화**: 기관별 내규까지 포함하여 일관된 계약 업무 지원
- **시스템 연동**: 모니터링 대시보드와 연계 → "이 계약이 왜 문제인지" 법적 근거 즉시 조회
